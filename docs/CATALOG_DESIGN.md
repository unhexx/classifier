# Дизайн справочника типовых неисправностей (Справочник классификации обращений)

> Основано на глубоком исследовании лучших практик крупных производственных компаний и IT-корпораций (2026).

## Цель справочника

Справочник служит источником истины для классификатора типовых неисправностей. Он позволяет по описанию симптомов (контекст обращения) находить наиболее вероятные неисправности и рекомендованные действия.

Ключевые требования к финальной версии:
- Профессиональное качество контента (реалистичные формулировки из практики эксплуатации).
- Структурированность по принципам индустрии.
- Высокая эффективность гибридного скорера (keywords + fuzzy + trigram).
- Поддержка дообучения через feedback контролёра.
- Все на русском языке, с практическими терминами (включая распространённые английские/лог-сообщения).

## Исследование: ключевые лучшие практики

### 1. ITIL / ITSM (Incident & Problem Management)
- Классификация иерархическая: Category (функциональная область) → Type (природа: Outage, Degradation, Error, Latency).
- Категории должны быть **функциональными**, а не привязанными к конкретным вендорам.
- Разделяют Reported Category и Resolution Category.
- Цели классификации: routing, trending, input в Problem Management и Knowledge Base.
- Важно: простота и usability — слишком детализированные таксономии не используются.

### 2. FMEA (Failure Mode and Effects Analysis) — IEC 60812, SAE J1739
- Структурированный подход: Component/Function → Failure Mode → Cause → Effect (локальный/системный) → Recommended Actions.
- Типичные Failure Modes (guidewords):
  - Full failure / No function
  - Partial / Degraded performance
  - Intermittent
  - Spurious / Unexpected operation
  - Failure to operate at prescribed time
  - Loss of function
  - High / Low (температура, давление, производительность)
- Используется в manufacturing, automotive, industrial для Reliability Centered Maintenance (RCM).

### 3. CMMS / EAM (IBM Maximo, SAP PM и аналоги)
- Иерархия Failure Codes: **Failure Class** → **Problem** → **Cause** → **Remedy**.
- Применяется на Work Order для накопления качественной истории отказов и Pareto-анализа.
- Лучшие практики:
  - Практичность (то, что реально используют техники).
  - Pre-populate на активах.
  - Эволюция на основе реальных данных.
  - Не делать слишком глубокую иерархию.

### 4. SRE (Google и крупные IT-компании)
- Классификация по **симптому + компоненту** + impact/severity.
- Акцент не столько на статическом справочнике, сколько на runbooks и postmortem (contributing factors).
- Полезно для нас: богатые симптомы, конкретные индикаторы (логи, метрики, визуальные признаки), привязка к действиям.

### 5. Общие выводы для нашего продукта
- Структура записи должна поддерживать:
  - Чёткую категорию (стандартизированную).
  - Failure Mode.
  - Богатые симптомы/keywords (логи, коды ошибок, индикаторы, разговорный язык).
  - Причинно-следственные связи + actionable recommended actions.
- Категории должны быть согласованы между доменами.
- Избегать "Other"/"Прочее".
- Контент должен быть живым: обновляться через admin + feedback loop.

## Стандартизированная таксономия (для всех каталогов)

### Рекомендуемые категории (стандартные)
1. **Питание** (Power, PSU, UPS, PoE)
2. **Хранение данных** (Storage, Disk, RAID, SAN, LUN, Backplane)
3. **Сетевое подключение / Связь** (Network Link, Interface, Transceiver)
4. **Охлаждение и вентиляция** (Cooling, Fans, Thermal)
5. **Процессор, память, контроллеры** (CPU, Memory, HBA, PCIe)
6. **Материнская плата / Аппаратная платформа** (Motherboard, BMC, POST)
7. **Конфигурация, прошивка и ПО** (Firmware, BIOS/UEFI, Config, Driver)
8. **Маршрутизация и L3** (Routing, BGP, OSPF, VLAN inter)
9. **L2 / Switching** (STP, LAG, VLAN, Port Security)
10. **Мониторинг, управление и сервисы** (BMC/iLO, SNMP, NTP, AAA, DNS)
11. **Производительность и деградация** (High CPU/Mem, Latency, Overload)
12. **Безопасность и доступ** (ACL, Certificate, Authentication)
13. **Виртуализация** (Hypervisor, VM, Datastore)
14. **Резервное копирование и восстановление** (Backup, Recovery)

Категории должны быть **консистентными** по всем справочникам.

### Поля записи (формат JSON)
Текущий формат остаётся базовым для совместимости. Рекомендуется расширять:

```json
{
  "code": "SRV-PSU-002",
  "title": "...",
  "description": "...",
  "failure_mode": "Полный отказ | Деградация | Интермиттент | ...",   // новое рекомендуемое поле
  "symptoms": ["..."],           // наблюдаемые проявления (что говорит пользователь/логи)
  "keywords": ["..."],           // для скорера (включая английские термины)
  "category": "Питание",
  "recommended_actions": ["..."],
  "meta": {
    "typical_cause": "...",
    "impact": "высокий | средний",
    "detection": "BMC, логи, визуально"
  }
}
```

**failure_mode** (рекомендуемые значения):
- Полный отказ (No function)
- Деградация / Частичный отказ
- Интермиттент (Intermittent)
- Неожиданное поведение (Spurious)
- Потеря производительности
- Ошибка конфигурации / прошивки

### Правила создания записей (guidelines)
- Code: `DOMAIN-XXX-NNN` (SRV-, NET-, AUTO-, IND-).
- Title: коротко и по делу (как в тикете).
- Description: 1-2 предложения, что именно происходит.
- Symptoms: 4–12 реальных фраз (логи, сообщения, индикаторы).
- Keywords: токены для Jaccard/fuzzy (включая распространённые ошибки, коды).
- Recommended actions: конкретные, пошаговые, с приоритетами и проверками.
- Избегать дубликатов. Каждая запись должна иметь уникальную комбинацию симптомов.
- Обогащать со временем через admin API + feedback.

## Стратегия наполнения

1. **Компонентный подход** (FMEA + CMMS).
2. **Покрытие самых частых причин отказов** по статистике железа (HDD/SSD, PSU, Fans, Link, Memory — топ в большинстве отчётов).
3. **Реалистичные примеры** из практики крупных компаний.
4. **Билингвизм** в keywords/symptoms (русские + английские лог-сообщения).
5. **Рекомендуемые действия** должны быть полезными для контролёра/инженера первой линии.

## План эволюции справочника
- Базовая версия (текущая) — в JSON seeds.
- Обогащение и расширение — через P1-08.
- Дальнейшее развитие — через Admin API + экспорт/импорт + анализ feedback.
- Валидация: обязательные поля + ограничение на длину + уникальность code внутри каталога.
